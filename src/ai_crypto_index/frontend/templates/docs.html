AI Crypto Index: weights, sleeves, and risk metrics

Daily refreshed sleeves with transparent run IDs and KPIs, ready to query through the API or SDKs.

The service generates daily and intraday index runs and returns deterministic allocations by run_id. Use one run_id across endpoints to keep weights, KPIs, and charts in sync.

TL;DR

Weights refresh after 07:00 UTC for daily mode. Each response includes a run_id in format YYYY-MM-DDTHH-MM-SSZ[-suffix] that you can pin across downstream calls.

Three sleeves are available: Classic (balanced growth), Conservative (lower volatility), and Aggressive (higher beta). All sleeves share one cleaned universe and the same run timestamp.

How it works

  1. Market quotes are ingested, quality filtered, and aligned into a consistent feature set.
  2. The model layer forecasts return and risk signals for each asset.
  3. The optimizer applies sleeve constraints and produces target weights.
  4. The result is stamped with run_id and published through API endpoints.

Output and reproducibility

  • /weights/latest returns the freshest run.
  • /runs/{run_id}/perf returns CAGR, volatility, max drawdown, and Sharpe for the same run.
  • /runs/{run_id}/mini returns equity-curve summary assets for charting.
  • Use one run_id end-to-end to avoid cross-run mismatch in reports.

Run parameters: theory, validation, impact

These UI parameters control universe size, history depth, forecast horizon, and risk constraints. Validation keeps combinations consistent. Stay within recommended ranges for stable results; move outside only for targeted experiments.

TL;DR

Defaults are designed for balanced, daily runs. Adjust with the tradeoffs below in mind and keep the validation rules intact.

Key tradeoffs

  • Diversification vs noise. Higher n_top_coins and total_assets reduce concentration risk but can add illiquid or noisy assets.
  • History vs responsiveness. Earlier start_date and larger lookback_days stabilize estimates but slow regime response.
  • Horizon vs adaptation. Higher forecast_horizon implies fewer weight changes; lower values adapt more often.
  • Constraints vs freedom. Tighter risk_min_weight, risk_max_weight, and weight_cap reduce concentration but can cap upside.
  • Forecast conservatism. advanced_forecast uses a richer model; higher gating_tolerance leans more conservative.
  • Risk buffer. Higher vol_floor_ratio inflates risk estimates and makes allocations more defensive.

Validation rules

  • risk_max_weight >= risk_min_weight.
  • total_assets <= n_top_coins.
  • risk_min_weight * total_assets <= 1.
  • risk_min_weight <= min(0.08, 1 / total_assets).
  • weight_cap >= max(0.08, 1 / total_assets) and weight_cap <= 0.30.
  • lookback_days >= window_size.
  • forecast_horizon <= window_size and forecast_horizon <= lookback_days.
  • gating_tolerance <= 0.10 (technical ceiling 0.25).
  • start_date is not in the future, not earlier than 2016-01-01, not later than today - max(lookback_days, 120d), and the available history covers lookback_days from 2015-01-01.

Plan limits. Free plan caps n_top_coins at 200 and total_assets at 12. Requests beyond the cap fall back to defaults.

Rate limits and retries

  • Edge cache keeps /weights/latest fast (around 120 ms p50 in internal tests).
  • Pipeline triggers charge base + parameter add-ons; status checks are free; run artifacts read is 5 tokens per request.
  • Indicative caps: Free up to 1k tokens/day and 5 rps, Pro up to 10k tokens/day and 40 rps, Enterprise negotiated.
  • Check X-RateLimit-* on every response and avoid tight retry loops.
429 / Retry-After. Pause for the Retry-After value before retrying. Add jittered backoff and avoid hammering /weights/latest; cache responses on your side for at least the 30s edge TTL.

FAQ: keys, limits, reliability

TL;DR

Keys are generated in the account dashboard (/app) under Keys. Keep keys server-side and pair them with AICI_BASE_URL. Use Retry-After on 429 and short delayed retries on transient 5xx responses.

Questions

How do I generate an API key?
Sign in to the account dashboard (/app), open Keys, and generate a key. Store it as AICI_API_KEY and never expose it in client-side code.
Which base URL should I use?
Use {{ base_api_url }} for production unless support provides a staging host. Keep URL in environment variables for easy switching.
What does run_id represent?
run_id is the UTC timestamp for one model build. Reuse it across endpoints to fetch consistent weights and metrics.
How does data freshness work for n_top_coins?
n_top_coins=100 uses the daily cached snapshot. Other values can trigger fresh snapshot retrieval and then reuse cache until expiration.
What are Free plan limits?
Free caps n_top_coins at 200 and final asset count at 12. Requests above limits fall back to defaults.
Why did I get 401 or 403?
Check that X-API-Key is present, active, and allowed for your source IP/domain, and ensure the request points to the correct base URL.
How should I handle 429?
Read Retry-After, apply exponential backoff with jitter, and avoid repeated hot-loop calls to /weights/latest.
What about 5xx or maintenance windows?
Retry after a short delay. If errors persist, contact support with your run_id and request ID.

Glossary

TL;DR

Index run is one timestamped build with a unique run_id. A sleeve is a risk profile. A benchmark is a reference portfolio used for comparison.

Terms

Index run
A full model build at a point in time, identified by run_id.
Sleeve
A risk profile and allocation mix: Classic, Conservative, or Aggressive.
Benchmark
A reference portfolio used to compare sleeve performance.
Weights
Asset allocations represented as decimals that sum to 1.0 for each sleeve.
Latency
API round-trip time between the client and endpoint response.
