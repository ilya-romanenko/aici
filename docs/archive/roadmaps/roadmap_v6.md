# Личный кабинет AICI — переход к структурированному UX

## Контекст
- Сейчас `/dashboard` и `/developers` обслуживают разные потребности без общей логики, что ломает целевой self‑serve сценарий из `stage0_self_serve_plan`.
- Пользователь не видит очевидный путь «регистрация → план → ключ → playground → мониторинг», нет чек-листов и статуса активации.
- Основные операции (API-ключи, биллинг, usage, документация) вывалены в одном полотне, поэтому теряется onboarding и апселл.

## Целевое решение
1. **Единый маршрут `/app`** c layout'ом «фиксированная левая панель + контентная область». Каждая вкладка имеет URL (`/app/overview`, `/app/keys`, `/app/billing`, `/app/usage`, `/app/playground`, `/app/support`), breadcrumbs и быстрые действия.
2. **Персональный hero-блок** на вкладке Overview: приветствие, статус воронки (Onboarding checklist + прогресс), план + лимиты, CTA «Создать ключ», «Запустить playground», уведомления.
3. **Группы функций**:
   - *Keys & Security*: список ключей, генерация/ротация, ограничения по IP/таймерам, webhooks активности.
   - *Billing & Plans*: тарифы, апгрейд/даунгрейд, история платежей, Stripe portal deep-link, промокоды.
   - *Usage & Alerts*: графики запросов, error-rate, квоты, настройка алертов и webhooks.
   - *Playground & Docs*: встроенный Web playground, интерактивные примеры, SDK-сниппеты, кнопка «Скопировать ключ».
   - *Support & Notifications*: тикеты, статусы email-рассылок, changelog продукта.
4. **UX-паттерны**: табы + вторичный вертикальный степпер для onboarding, контекстные подсказки, сохранённые фильтры, а также пустые состояния для новых пользователей.
5. **Технический подход**: переиспользовать существующий FastAPI + Jinja стек, обернув новый layout компонентом `account_base.html`, выделить API для usage и биллинга, добавить фронтовые модули (HTMX или SPA, если решат мигрировать позже).
