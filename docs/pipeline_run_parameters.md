# Справочник параметров запуска пайплайна

Документ описывает параметры запуска пайплайна (API `POST /api/v1/run` и `POST /api/v1/run/async`), их дефолты, места применения, а также влияние на списание токенов, риск и доходность.

## Где задаются параметры

- Модель запроса и дефолты: `src/ai_crypto_index/api/models.py` (`RunRequest`).
- Query-параметры API: `src/ai_crypto_index/api/app.py` (`_collect_run_query_overrides`).
- Исполнение пайплайна: `src/ai_crypto_index/pipelines/main.py` (`run_monthly_update`, `compute_monthly_weights_for_df`).
- Лимиты free-плана: `src/ai_crypto_index/api/app.py` (`_apply_free_plan_run_limits`).

## Токены и биллинг

- Стоимость вызова пайплайна считается как базовая цена из `config/pipeline.json` → `api_keys.token_pricing.pipeline_trigger_tokens` (по умолчанию 2) **плюс** параметрические надбавки (см. ниже). Применяется в `src/ai_crypto_index/api/app.py` при `POST /api/v1/run*`.
- Статусные проверки по run (прогресс и `/runs/{run_id}/result` пока нет данных) бесплатны; получение артефактов (`/weights/latest`, `/runs/{run_id}/weights`, `/runs/{run_id}/perf`, `/runs/{run_id}/export` и `/runs/{run_id}/result` с данными) списывает **5 токенов**.
- Значения параметров запуска меняют списание токенов через параметрические надбавки. Итоговая стоимость округляется вверх до целого.

### Параметрическое списание

Надбавки считаются поверх базовой стоимости вызова (`pipeline_trigger_tokens`).

- `n_top_coins`: +`0.5` токена за монету (вариант смягчения: первые 50 монет без доплаты).
- `advanced_forecast=True`: +`10` токенов за актив в `total_assets`.
- `advanced_forecast=False`: +`4` токена за актив в `total_assets`.
- Кастомный `run_id`: +`5` токенов за запуск (только если `run_id` передан пользователем).
- `visualization=True`: +`5` токенов за запуск.
- Длинная история (`lookback_days`/`start_date`): доплата `max(0, ceil((history_days - 365) / 90))` токенов, где `history_days` = разница между `today` и `start_date` (если задан) либо `lookback_days`.

## Валидация и автокоррекции

Источник: `src/ai_crypto_index/api/models.py` и `src/ai_crypto_index/api/app.py`.

- `risk_max_weight >= risk_min_weight`.
- `total_assets <= n_top_coins`.
- `risk_min_weight * total_assets <= 1`.
- `risk_min_weight <= min(0.08, 1 / total_assets)`.
- `weight_cap >= max(0.08, 1 / total_assets)` и `weight_cap <= 0.30`.
- `lookback_days >= window_size`.
- `forecast_horizon <= window_size` и `forecast_horizon <= lookback_days`.
- `gating_tolerance <= 0.10` (технический потолок `0.25`).
- `start_date` не в будущем, не раньше `2016-01-01`, не позже `today - max(lookback_days, 120d)`, плюс окно `lookback_days` не должно выходить за границу доступной истории с `2015-01-01`.
- Free план: `n_top_coins <= 200`, `total_assets <= 12`; при превышении значения сбрасываются на дефолты.

## Рекомендуемые лимиты (используются в валидации)

Ниже — ориентировочные границы, вне которых параметры часто дают деградацию качества или заметно удорожают/замедляют запуск.

| Параметр | Мин (soft) | Макс (soft) | Комментарий |

| `start_date` | 2016-01-01 | не позже `today - max(lookback_days, 120d)` | Нужен достаточный хвост истории; слишком старая дата тянет редкие/шумные данные. |
| `lookback_days` | 90 | 720 | Меньше 3 мес. дает шум, больше 2 лет снижает реактивность. |
| `window_size` | 14 | 120 | Должно быть `<= lookback_days`. |
| `forecast_horizon` | 7 | 60 | Должно быть `<= window_size` и `<= lookback_days`. |
| `advanced_forecast` | н/д | н/д | Булевый флаг. |
| `fresh_data` | н/д | н/д | DEV_ONLY: устарел, игнорируется, не в UI/playground. |
| `info_messages` | н/д | н/д | DEV_ONLY: не в UI/playground, значение игнорируется. |
| `visualization` | н/д | н/д | DEV_ONLY: не в UI/playground, значение игнорируется. |

| `clustering_metric` | н/д | н/д | Сейчас не влияет на расчеты. |
| `risk_min_weight` | 0.005 | `min(0.08, 1 / total_assets)` | Должно быть `risk_min_weight * total_assets <= 1`. |

| `weight_cap` | `max(0.08, 1 / total_assets)` | 0.30 | Слишком низкий cap конфликтует с фиксированным `total_assets`. |

## Параметры запуска

Параметры, помеченные `DEV_ONLY`, не используются в UI app/playground и должны быть исключены из пользовательской документации.

### n_top_coins
- Дефолт: `100`.
- Где используется: выбор универсума при загрузке/снапшоте данных (`daily_snapshot.select_snapshot_dataframe`, `prepare_environment`).
- Токены: не влияет напрямую.
- Риск/доходность: увеличение расширяет вселенную и диверсификацию, но может добавить более волатильные/низколиквидные монеты.
- Примечания: лимит free-плана — `<= 200`.

### start_date
- Дефолт: `None` (в `run_monthly_update` подставляется «год назад»).
- Где используется: обрезка исторических данных перед расчетами.
- Токены: не влияет напрямую.
- Риск/доходность: длинная история сглаживает шум и снижает риск переобучения, но может ухудшить реактивность; короткая история делает модель более «быстрой», но повышает риск.
- Примечания: не раньше `2016-01-01`.

### lookback_days
- Дефолт: `180`.
- Где используется: окно данных для расчета лог-доходностей и отбора активов.
- Токены: не влияет напрямую.
- Риск/доходность: большее окно снижает чувствительность к режимам, меньшее делает портфель более агрессивным.
- Примечания: должно быть `>= window_size`.

### window_size
- Дефолт: `30`.
- Где используется: размер окна LSTM/EWMA, влияние на прогноз и построение волатильности.
- Токены: не влияет напрямую.
- Риск/доходность: меньшее окно повышает шум и риск; большее окно сглаживает, но снижает реактивность.

### forecast_horizon
- Дефолт: `30`.
- Где используется: горизонт прогноза волатильности и периода применения весов.
- Токены: не влияет напрямую.
- Риск/доходность: более длинный горизонт повышает модельный риск, более короткий — повышает частоту адаптации.
- Примечания: `<= window_size` и `<= lookback_days`.

### advanced_forecast
- Дефолт: `True`.
- Где используется: переключение режима прогнозов (LSTM + gating) или базовый EWMA.
- Токены: не влияет напрямую.
- Риск/доходность: LSTM потенциально повышает доходность, но несет риск переобучения; EWMA более консервативен.

### fresh_data (DEV_ONLY)
- Дефолт: `False`.
- Где используется: **сейчас игнорируется**; параметр объявлен устаревшим в API и в пайплайне.
- Токены: не влияет.
- Риск/доходность: не влияет.

### info_messages (DEV_ONLY)
- Дефолт: `False`.
- Где используется: сейчас не используется в UI и в API; значение игнорируется.
- Токены: не влияет.
- Риск/доходность: не влияет.

### visualization (DEV_ONLY)
- Дефолт: `False`.
- Где используется: сейчас не используется в UI и в API; значение игнорируется.
- Токены: не влияет напрямую.
- Риск/доходность: не влияет; может увеличить время выполнения при возвращении визуализаций.

### total_assets
- Дефолт: `10`.
- Где используется: число активов, выбираемых после кластеризации.
- Токены: не влияет напрямую.
- Риск/доходность: больше активов = выше диверсификация и ниже концентрационный риск; меньше активов = выше шанс на альфу и выше риск.
- Примечания: `<= n_top_coins`, лимит free-плана — `<= 12`.

### clustering_metric
- Дефолт: `"sharpe"`.
- Где используется: передается в `select_assets_balanced`, **но сейчас не используется внутри функции**.
- Токены: не влияет.
- Риск/доходность: сейчас не влияет.

### risk_min_weight
- Дефолт: `0.03`.
- Где используется: нижняя граница при расчете risk parity.
- Токены: не влияет.
- Риск/доходность: выше минимум = более равномерный портфель и ниже риск концентрации; ниже минимум = больше свободы для перекосов.
- Примечания: `risk_min_weight * total_assets <= 1`.

### risk_max_weight
- Дефолт: `0.25`.
- Где используется: верхняя граница при risk parity.
- Токены: не влияет.
- Риск/доходность: ниже максимум = ниже риск концентрации, но может ограничить доходность; выше максимум = больше концентрации.
- Примечания: `risk_max_weight >= risk_min_weight`.

### weight_cap
- Дефолт: `0.15`.
- Где используется: пост-обрезка весов после risk parity с нормализацией.
- Токены: не влияет.
- Риск/доходность: ниже cap = более равномерное распределение; выше cap = допускает концентрацию.

### vol_floor_ratio
- Дефолт: `0.4`.
- Где используется: нижняя граница прогнозной волатильности относительно исторической.
- Токены: не влияет.
- Риск/доходность: выше значение делает портфель более консервативным (меньше недооценки риска), ниже — допускает более агрессивные веса.

### gating_tolerance
- Дефолт: `0.02`.
- Где используется: порог улучшения LSTM относительно наивной модели в `gate_and_mix`.
- Токены: не влияет.
- Риск/доходность: выше толеранс = больше EWMA (консервативнее), ниже = больше LSTM (более агрессивно).
- Примечания: влияет только при `advanced_forecast=True`.

### run_id
- Дефолт: `None` (автогенерация).
- Где используется: имя директории прогона, кэш результатов, прогресс.
- Токены: +`5` за пользовательский `run_id` (если передан явно).
- Риск/доходность: не влияет.

## Связанные настройки окружения (не параметры run)

Эти поля из `config/pipeline.json` влияют на данные и подготовку, но не являются параметрами запуска:
- `data.*` — фильтрация данных (dropna, min_history_days, include_delisted).
- `market_data.top_n`, `market_data.start_date`, `market_data.fresh_download` — загрузка данных при подготовке окружения/снапшотов.
- `runs.*` — расположение артефактов прогона.
